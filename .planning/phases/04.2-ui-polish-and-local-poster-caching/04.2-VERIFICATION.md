## Gaps Summary

Three gaps block full phase goal achievement:

**Gap 1 — Poster wall not displaying (truth 1, FAILED)**
The PosterWall component and all supporting infrastructure (CSS keyframes, Tailwind animations, StaticFiles mount, compose.yaml volume, nightly download pass) are correctly implemented. The runtime failure on NAS is traced to the `GET /movies/poster-wall` endpoint returning fewer than 5 items with a 6-movie watch history. The most likely cause is that the WatchEvent-to-Movie JOIN has a data gap: watched movies may not have `poster_path` populated in the movies table, or the WatchEvent records reference movie IDs not yet fully hydrated. Investigation requires running the endpoint live on NAS. The poster wall's TMDB CDN fallback in `posterUrl()` should theoretically allow display even before local download — this makes the root cause more urgent to diagnose.

**Gap 2 — Random Pick button permanently disabled (truth 3, FAILED)**
This is a confirmed logic bug in `GameSession.tsx` line 676. The disabled condition `(filteredMovies.length === 0 || !selectedActor)` prevents the button from activating when the user browses movies in combined view (no actor selected) or after clicking "Show all eligible movies". The fix is to remove `!selectedActor` from the disabled condition — Random Pick should work whenever `filteredMovies.length > 0`, and the dialog's "Request This Movie" handler already handles the case where `selectedActor` is null (it only calls `api.pickActor` when `selectedActor` is truthy).

**Gap 3 — "Started" stat shows negative time (truth 2, PARTIAL)**
The `formatSessionAge` function in `SessionCounters.tsx` computes `Date.now() - new Date(createdAt).getTime()`. When the backend serializes a naive UTC datetime (no `Z` suffix), JavaScript's `Date` constructor on a UTC-7 machine treats the string as local time, making the session appear to be 7 hours in the future. The fix can be applied in either the frontend (append `Z` if not present before parsing) or the backend (serialize `created_at` with a `Z` suffix by appending `isoformat() + "Z"` or using a timezone-aware datetime type). Given `-420min ago` is exactly -7 hours, this is definitively a UTC/local timezone mismatch.
