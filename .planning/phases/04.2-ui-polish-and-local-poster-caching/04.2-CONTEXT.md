# Phase 4.2: UI Polish & Local Poster Caching — Context

**Gathered:** 2026-03-17
**Status:** Ready for planning

<domain>
## Phase Boundary

Elevate the visual experience with a drifting poster-wall background sourced from the user's watched movie history, add quality-of-life game controls (random pick button, dead-end detection), expand session stats to two locations, and eliminate runtime dependency on TMDB CDN by caching poster images locally during the nightly sync.

No new game mechanics. No bug fixes. Enhancement and polish only.

</domain>

<decisions>
## Implementation Decisions

### Poster-Wall Background

- **Source:** Watched movies across all sessions (`WatchEvent` records → `Movie.poster_path`). If fewer than 20 posters exist in DB, supplement with random popular movies fetched from TMDB (`/movie/popular`) until 20 are available.
- **Style:** 3–4 vertical columns of poster images that drift at different speeds in alternating directions (columns 1 & 3 scroll up, columns 2 & 4 scroll down). CSS infinite animation (`@keyframes`), no JS scroll logic needed.
- **Coverage:** Full-page behind the entire `GameSession` page — session home view and tab view (actors/movies/suggested). One consistent cinematic backdrop across the whole session experience.
- **Blur + overlay:** Heavy blur (`blur(20px)`) + dark overlay (`rgba(0, 0, 0, 0.80)`). Posters are felt as texture — not readable. UI cards sit on standard `bg-card` without needing additional frosted-glass treatment.
- **Poster size:** Use local cached poster paths when available (Phase 4.2 backend work), fall back to `https://image.tmdb.org/t/p/w185{poster_path}`. Small image size is appropriate — heavy blur means resolution doesn't matter.
- **Fallback when no poster:** If a movie has no `poster_path`, skip it when building the column list. If fewer than 5 posters are available total, show the standard `bg-background` (no animated wall).
- **Animation speed:** Claude's discretion — aim for slow drift (~60–90 seconds per full cycle) so it's atmospheric rather than distracting.

### Random Pick Button

- **Scope:** Eligible Movies tab only. Requires the user to have already selected an actor (same precondition as the existing movie grid).
- **Placement:** Button in the Eligible Movies tab header row, alongside the tab title. Icon-only or icon+label ("Random") — Claude's discretion on exact label.
- **Behavior:** Picks a random movie from the current eligible list (respecting active filters), then shows a shadcn Dialog:
  - Title: "Random Pick"
  - Body: movie title (and poster thumbnail if available)
  - Actions: [Cancel] / [Confirm]
- **On confirm:** Same flow as a normal movie selection (calls `request_movie`, advances chain, shows Radarr notification).
- **When no eligible movies exist:** Button is disabled (greyed out). No dialog shown.

### Dead-End Detection

- **Trigger:** When the Eligible Actors tab loads with zero results (all actors in the current movie's cast have already been used in this session).
- **UX:** Replace the empty actors grid with a clear empty state message. No proactive warning before the user clicks "Continue the chain" — detection is reactive (shown when they open the tab).
- **Empty state content:**
  - Heading: "Chain dead end"
  - Body: "All actors from this movie have been used in this session."
  - Two action buttons: [Delete Last Step] (calls existing delete-last-step flow) and [End Session]
- **No tab badge or visual indicator on the session home page** — the dead end is only surfaced when they attempt to pick actors.

### Session Stats — Two Locations

**New stats to add** (in addition to existing "movies watched" and "total runtime"):
1. **Chain length** — number of steps taken (actor picks made)
2. **Unique actors** — count of distinct actors used across all steps
3. **Session duration** — time elapsed since `created_at` (displayed as "Xh Ym ago" or "Xd ago")

**Location 1 — SessionCounters component (session header):**
- Expand the existing `SessionCounters` bar to include the three new stats alongside the existing two.
- All five stats in one scannable row: movies watched · total runtime · chain steps · unique actors · started X ago

**Location 2 — Session cards on GameLobby home page:**
- The active session cards in the lobby grid currently show session name and status.
- Add a compact stats line below the session name: movies watched, chain steps, started X ago.
- Unique actors and total runtime are secondary — include if space allows; Claude's discretion on which subset fits cleanly in the card layout.

### Local Poster Caching (Nightly Sync)

- **Mechanism:** During the nightly TMDB cache job, for each `Movie` row where `poster_path IS NOT NULL` and no local file exists yet, download the poster from `https://image.tmdb.org/t/p/w185{poster_path}` and save to `/static/posters/{tmdb_id}.jpg`.
- **DB column:** Add `poster_local_path VARCHAR(255)` (nullable) to the `Movie` model. Alembic migration required.
- **Frontend preference:** When building poster URLs, prefer `/static/posters/{tmdb_id}.jpg` (local) and fall back to the TMDB CDN URL if `poster_local_path` is null. This fallback covers new movies not yet synced.
- **Static file serving:** FastAPI must serve `/static/` via `StaticFiles` mount pointing to the local posters directory. The directory must be volume-mounted in Docker Compose so it persists across container restarts.
- **Cache invalidation:** No active invalidation. Re-download only if the `poster_path` value changes (detectable by re-sync diff). Posters are effectively immutable once fetched.
- **Scope of nightly download:** All movies already in the DB with a `poster_path`. New movies added by future syncs get their poster downloaded on the next nightly run.

### Claude's Discretion
- CSS animation speed for poster columns (target: slow and atmospheric, ~60–90s cycle)
- Column count (3 or 4 columns depending on viewport)
- Exact random pick button label and icon
- Subset of stats shown on GameLobby session cards (space-dependent)
- Docker volume name and host path for the static posters directory

</decisions>

## Existing Code Insights

### Reusable Assets
- `SessionCounters.tsx`: existing component with watched count + runtime — extend, don't replace
- `GameSession.tsx` root div: `min-h-screen bg-background flex flex-col` — the `bg-background` class is replaced by the poster-wall; position the column container as `fixed inset-0 -z-10` beneath the existing layout
- `currentStep?.poster_path` already computed in `GameSession.tsx` — poster URL pattern established
- `cache.py` nightly job: `_ensure_movie_details_in_db` already downloads movie metadata; poster download is a new pass in the same nightly loop
- `WatchEvent` model: source of watched movie history for poster wall population
- Delete-last-step flow: already implemented (Phase 4); dead-end empty state links directly to it

### Established Patterns
- TMDB image URL: `https://image.tmdb.org/t/p/w185{poster_path}` — use same tier for poster wall (resolution hidden by blur)
- shadcn Dialog: used for delete confirmations; use same component for random pick confirm dialog
- React Query: `useQuery` pattern established throughout — poster wall data fetched via new query at `GameSession` mount
- `apiFetch` wrapper in `api.ts`: all API calls go through this; new `/sessions/{id}/poster-wall` or a dedicated `/movies/poster-wall` endpoint follows same pattern

### Integration Points
- Poster wall data: new backend endpoint `GET /movies/poster-wall?limit=40` — returns list of `{tmdb_id, poster_path, poster_local_path}` from WatchEvents; supplements with TMDB popular if <20 results
- `StaticFiles` mount: `app.mount("/static", StaticFiles(directory="static"), name="static")` added to `main.py`; Docker Compose adds volume `./static:/app/static`
- GameLobby session cards: `listSessions` API already returns session data; backend must include new stats fields (`step_count`, `unique_actor_count`, `created_at`) in `GameSessionDTO`
