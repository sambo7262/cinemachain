# Phase 03.2: Game UX Enhancements — Context

**Gathered:** 2026-03-16
**Status:** Ready for planning

<domain>
## Phase Boundary

Enrich the in-session gameplay experience: faster movie selection via filters and search, richer chain history display with thumbnails, session progress counters, and an ineligible actor reveal toggle. Does NOT change game rules, session lifecycle, or the data model for sessions.

</domain>

<decisions>
## Implementation Decisions

### Movie Filters — Layout
- Filters live in a **collapsible sidebar** on the eligible-movies panel
- Each filter category is independently collapsible (collapsed by default, expand on click)
- Three filters to implement:
  1. **Runtime** — range slider (min/max minutes)
  2. **Genre** — multi-select checkboxes (genres sourced from `Movie.genres` JSON field, already in DB)
  3. **MPAA Rating** — checkbox group: G, PG, PG-13, R, NR (Not Rated / unknown)
- Filters apply client-side to the loaded eligible-movies list (no extra API calls per filter change)

### Movie Title Search
- A search input above the eligible-movies list filters the currently loaded list client-side — no backend call
- Filters as you type (instant, debounced ~200ms)
- Works in conjunction with sidebar filters (both apply simultaneously)

### TMDB Rating — Vote Floor
- When sorting eligible movies by TMDB rating (`vote_average`), movies with **fewer than 500 votes** are treated as unrated and sorted to the bottom
- `vote_count` is a new field that must be stored on the `Movie` model and returned in the eligible-movies response
- `vote_count` is populated from the TMDB credits/movie-details response (already fetched during `_ensure_actor_credits_in_db`) — store it alongside `vote_average`

### MPAA Rating — Data Source
- MPAA certification (US rating: G, PG, PG-13, R, NC-17, NR) is fetched **on-demand** from TMDB `/movie/{id}/release_dates` the first time a movie appears in an eligible-movies response
- Result stored as a new `mpaa_rating: str | None` column on the `Movie` model
- After first fetch it is cached — no re-fetch on subsequent requests
- If TMDB returns no US certification, store as `None` and display as "NR" in the filter UI

### Ineligible Actor Toggle
- A toggle button on the eligible-actors panel reveals actors who are ineligible (already picked in this session)
- Ineligible actors appear in a **separate section below** the eligible list, visually muted (lower opacity, "Already picked" badge)
- Backend: add `include_ineligible=true` query param to `GET /sessions/{id}/eligible-actors`; returns all cast members with an `is_eligible: bool` flag on each
- Default (no param): existing behavior — only eligible actors returned

### Chain History — Thumbnails
- Movie rows in the chain history table show a **movie poster thumbnail** (`poster_path` from TMDB, stored on `Movie` model)
- Actor rows show an **actor profile thumbnail** (`profile_path` from TMDB, stored on `Actor` model)
- Both paths already exist in the DB; backend must JOIN them into the step response (add `poster_path` and `profile_path` to `GameSessionStepDTO`)
- Thumbnails are small (movie: ~40×56px poster ratio; actor: ~28×28px circle)

### Session Counters
- Session page shows two counters near the top:
  1. **Movies watched** — count of steps where `watched_at IS NOT NULL`
  2. **Total runtime** — sum of `Movie.runtime` for watched steps only (steps with `watched_at IS NOT NULL` and `actor_tmdb_id IS NULL`)
- Runtime displayed as `Xh Ym` (e.g. "4h 22m"); if runtime data is missing for any movie, that movie's runtime is excluded from the sum (no placeholder)
- Backend must return these aggregates in `GameSessionResponse` — two new fields: `watched_count: int` and `watched_runtime_minutes: int`

### Chain History — Position
- Chain history table remains at the **bottom of the session page**, below the main action area (consistent with Phase 03.1 decision)

### Claude's Discretion
- Exact sidebar layout, filter chip styling, and expand/collapse animation
- Whether genre filter options are sorted alphabetically or by frequency
- Poster/profile image fallback when `poster_path`/`profile_path` is null (grey placeholder)
- Runtime slider step increments (e.g. 15min steps)
- Exact visual treatment of "Already picked" actors in the ineligible section

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `Movie.genres`: already stored as JSON-encoded string (e.g. `'["Action", "Drama"]'`) — parse with `json.loads()` for filter options
- `Movie.runtime`, `Movie.vote_average`, `Movie.poster_path`: all present — no migration needed for these fields
- `Actor.profile_path`: present in `Actor` model and already returned in eligible-actors response
- `EligibleMovieDTO` (api.ts): has `poster_path`, `vote_average`, `genres`, `runtime` — add `vote_count` and `mpaa_rating`
- Tailwind + shadcn/ui components already in frontend — use `Slider`, `Checkbox`, `Collapsible` components for filter sidebar

### Established Patterns
- TMDB on-demand fetch pattern: `_ensure_actor_credits_in_db` shows the fetch-and-cache pattern to replicate for MPAA
- Sort is currently applied server-side in `eligible-movies`; filters can be applied client-side on the already-fetched list
- `_build_session_response` is the central place to add new session-level aggregates (`watched_count`, `watched_runtime_minutes`)

### Integration Points
- New DB migration needed: add `vote_count: int | None` and `mpaa_rating: str | None` to `Movie` model
- `eligible-actors` endpoint: add `include_ineligible` query param + `is_eligible` flag on each actor in response
- `GameSessionStep` responses need `poster_path` and `profile_path` — join from `Movie` and `Actor` tables in `_build_session_response` or `_enrich_steps_watched_at`

</code_context>

<specifics>
## Specific Requirements

- Vote floor confirmed: **500 votes** minimum for TMDB rating sort to be applied
- Sidebar filters: collapsed by default, expand individually
- Search: filters loaded list instantly (client-side), not a new backend query
- MPAA: on-demand fetch, cached after first retrieval, `None` displayed as "NR"
- Ineligible actors: separate section below eligible list, not interleaved
- Runtime counter: watched movies only, format `Xh Ym`

</specifics>

<deferred>
## Deferred Ideas

- Paginating the eligible-movies list (currently loads all at once — fine for now)
- TMDB rating as a filterable range (user excluded this from Phase 03.2 scope; vote floor covers the main annoyance)
- Rotten Tomatoes scores in movie details (no public API exists)

</deferred>
