# Phase 03.2 Plan 30: Fix _ensure_movie_cast_in_db SP-1 Regression Summary

**One-liner:** Fixed three bugs in `_ensure_movie_cast_in_db` that caused South Park (and any animated film with multi-role voice actors) to return 0 eligible actors: missing `fetched_at` on Actor/Movie pg_inserts, and `credits.character` not truncated to VARCHAR(255).

## What Was Done

Diagnosed via `docker logs cinemachain-backend` after NAS deploy — error was `StringDataRightTruncationError: value too long for type character varying(255)` on the Credit insert.

Three fixes applied to `_ensure_movie_cast_in_db` (game.py lines 375–422):

1. **`pg_insert(Movie)`** — added `fetched_at=_datetime.utcnow()`. Column is `NOT NULL` in the migration (`nullable=False`, no `server_default`). Raw `pg_insert` bypasses ORM Python-side defaults, causing a NOT NULL violation on new Movie stubs.

2. **`pg_insert(Actor)`** — same fix. `actors.fetched_at` has the same constraint. In the background path the exception was swallowed silently; in the on-demand path it propagated as a 500. The `fetched_at` bug was the original SP-1 root cause hypothesis — confirmed.

3. **`credits.character[:255]`** — TMDB concatenates all roles a voice actor plays into a single character string (e.g. Trey Parker: `"Stan Marsh / Eric Cartman / Gregory / Satan / Mr. Garrison / ..."` — 300+ chars). Truncated at 255 before insert.

4. **`on_conflict_do_nothing(index_elements=["movie_id", "actor_id"])`** — Credit unique constraint is `(movie_id, actor_id)`; without `index_elements` PostgreSQL raises an error if multiple constraints exist.

Root cause sequence: `fetched_at` NOT NULL → exception in on-demand path → transaction invalidated → 0 actors returned. `character` overflow was hit immediately after the `fetched_at` fix landed because the Actor/Movie inserts then succeeded and reached the Credit insert.

## Verification

NAS deploy confirmed: South Park: Bigger Longer Uncut now returns eligible actors. SP-1 resolved. Full regression verified in 03.2-31.
