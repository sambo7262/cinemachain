## Summary

Phase 03.2 adds six independent UX improvements to the game loop: movie filters (runtime, genre, MPAA), title search, vote-floor sort fix, ineligible actor toggle, chain history thumbnails, and session counters. The work splits cleanly into backend model/migration changes and frontend UI additions. No game rules, session lifecycle, or session data model structure changes.

All decisions are locked. The research confirms every proposed implementation path is sound given the existing codebase. The main novelties are: a new Alembic migration (0005) adding `vote_count` and `mpaa_rating` to `Movie`; an on-demand TMDB fetch for MPAA ratings following the established `_ensure_actor_credits_in_db` pattern; client-side filtering and debounced search in the frontend; and JOIN enrichment to carry `poster_path`/`profile_path` through `StepResponse`/`GameSessionStepDTO`.

**Primary recommendation:** Implement in three waves ŌĆö (1) backend migration + model changes, (2) backend endpoint changes (eligible-actors `include_ineligible`, eligible-movies `vote_count`/`mpaa_rating`, MPAA on-demand fetch, session counters, step thumbnails), (3) frontend changes (filters/search sidebar, thumbnails in ChainHistory, counters display).

### Pattern 2: MPAA On-Demand Fetch (mirrors `_ensure_actor_credits_in_db`)

The established pattern for on-demand TMDB data that is cached after first fetch:

```python
# In get_eligible_movies, after building movies_map:
# For each movie that lacks mpaa_rating, fetch from TMDB and store
for tmdb_id, movie_dict in movies_map.items():
    if movie_dict.get("mpaa_rating") is None:
        # fetch and cache
        r = await tmdb._client.get(f"/movie/{tmdb_id}/release_dates")
        # extract US certification, update Movie row, update movies_map
```

Key details:
- TMDB `/movie/{id}/release_dates` returns `results` list; filter `iso_3166_1 == "US"`; take first `release_dates[].certification` that is non-empty
- Store result (or `""` meaning "checked but empty") vs. `None` meaning "never checked"
- Use `""` as sentinel for "checked, no US rating found" so re-fetch is not triggered again ŌĆö display as "NR"
- SQLAlchemy update: `await db.execute(sa.update(Movie).where(Movie.tmdb_id == tmdb_id).values(mpaa_rating=cert))`

**WARNING ŌĆö N+1 risk:** The eligible-movies list can be large (50-200 movies). Fetching MPAA for each uncached movie sequentially will be slow. Mitigate: batch the TMDB calls using `asyncio.gather` with a semaphore (existing TMDB rate-limit pattern). For a first implementation, sequential is acceptable on NAS because MPAA is cached after first fetch ŌĆö cold start is the only slow path.

### Pattern 3: vote_count in `_ensure_actor_credits_in_db`

The `cast` items from `/person/{id}/movie_credits` include `vote_count`. Currently the upsert in `_ensure_actor_credits_in_db` does not store it:

```python
# Current:
movie_stmt = pg_insert(Movie).values(
    tmdb_id=credit_data["id"],
    ...
    vote_average=credit_data.get("vote_average"),
    genres=None,
).on_conflict_do_nothing(...)
```

Change to `on_conflict_do_update` for `vote_count` (and `vote_average` since it may be stale):
```python
.on_conflict_do_update(
    index_elements=["tmdb_id"],
    set_={"vote_count": credit_data.get("vote_count"), "vote_average": credit_data.get("vote_average")}
)
```

**Note:** `on_conflict_do_nothing` must stay for most fields to avoid overwriting richer data. Only update the specific fields being enriched.

### Pattern 5: Step Thumbnails via JOIN

`StepResponse` currently derives data from `GameSessionStep` only. Both `poster_path` (Movie) and `profile_path` (Actor) need to come from their respective tables.

Two options:
1. **JOIN in `_build_session_response`** ŌĆö fetch Movie and Actor rows for steps at build time
2. **Separate query in `_enrich_steps_watched_at` analog** ŌĆö a new `_enrich_steps_thumbnails` helper

Option 2 is cleaner and follows the `_enrich_steps_watched_at` pattern exactly:

```python

    """Return (movie_tmdb_id->poster_path, actor_tmdb_id->profile_path)."""
    movie_tmdb_ids = [s.movie_tmdb_id for s in steps]
    actor_tmdb_ids = [s.actor_tmdb_id for s in steps if s.actor_tmdb_id]
    # Two selects ŌĆö simple, no complex JOIN
    ...
```

`StepResponse` gains two new nullable fields: `poster_path: str | None = None` and `profile_path: str | None = None`.

`_build_session_response` gains a `thumbnail_map` parameter alongside `watched_at_map`.

### Pattern 6: Session Counters in `_build_session_response`

Two new fields on `GameSessionResponse`:
```python
watched_count: int = 0
watched_runtime_minutes: int = 0
```

Computed in `_build_session_response` from the existing `watched_at_map` and a new `runtime_map`:

```python
# watched_count: number of movie steps (actor_tmdb_id IS NULL) with watched_at
watched_count = sum(
    1 for s in session.steps
    if s.actor_tmdb_id is None and (watched_at_map or {}).get(s.movie_tmdb_id)
)
# watched_runtime_minutes: sum of runtime for watched movie steps
watched_runtime_minutes = sum(
    runtime_map.get(s.movie_tmdb_id) or 0
    for s in session.steps
    if s.actor_tmdb_id is None and (watched_at_map or {}).get(s.movie_tmdb_id)
)
```

`runtime_map` is a new helper (similar to `_enrich_steps_watched_at`) that fetches `Movie.runtime` for all movie tmdb_ids in steps.

### Pattern 7: Client-Side Filtering (Frontend)

All eligible movies are already loaded into `allMovies` state. Filtering is pure JavaScript ŌĆö no additional API calls.

Filter state:
```typescript
const [searchQuery, setSearchQuery] = useState("")
const [debouncedSearch, setDebouncedSearch] = useState("")
const [genreFilter, setGenreFilter] = useState<string[]>([])
const [mpaaFilter, setMpaaFilter] = useState<string[]>([])  // ["G","PG","PG-13","R","NR"]
const [runtimeRange, setRuntimeRange] = useState<[number, number]>([0, 300])
```

Debounce pattern (no library):
```typescript
useEffect(() => {
  const t = setTimeout(() => setDebouncedSearch(searchQuery), 200)
  return () => clearTimeout(t)
}, [searchQuery])
```

Filter pipeline applied to `eligibleMovies` before rendering:
```typescript
const filteredMovies = eligibleMovies
  .filter(m => debouncedSearch === "" || m.title.toLowerCase().includes(debouncedSearch.toLowerCase()))
  .filter(m => genreFilter.length === 0 || parseGenres(m.genres).some(g => genreFilter.includes(g)))
  .filter(m => mpaaFilter.length === 0 || mpaaFilter.includes(m.mpaa_rating ?? "NR"))
  .filter(m => {
    if (m.runtime == null) return true  // include movies with unknown runtime
    return m.runtime >= runtimeRange[0] && m.runtime <= runtimeRange[1]
  })
```

Genre options are derived from the loaded movie list: `[...new Set(eligibleMovies.flatMap(m => parseGenres(m.genres)))]`.

### Pattern 8: Vote Floor Sort

Current sort in `get_eligible_movies`:
```python
movies.sort(key=lambda m: (m["vote_average"] is None, -(m["vote_average"] or 0)))
```

With vote floor (500 votes):
```python
VOTE_FLOOR = 500

def _effective_rating(m: dict) -> float | None:
    """Return vote_average only if vote_count >= VOTE_FLOOR, else None."""
    if m.get("vote_count") is None or m["vote_count"] < VOTE_FLOOR:
        return None
    return m.get("vote_average")

movies.sort(key=lambda m: (_effective_rating(m) is None, -(_effective_rating(m) or 0)))
```

### Anti-Patterns to Avoid

- **Re-fetching eligible movies on every filter change:** Filters are client-side. The existing query (`eligibleMovies`) is fetched once per actor selection. Do not add new query params for filter values.
- **Storing mpaa_rating="" vs. None ambiguity:** Use `None` = never fetched; non-None string (including `""`) = fetched. Display `""` as "NR" in UI. This prevents re-fetch loops.
- **Storing vote_count with `on_conflict_do_nothing`:** The existing upsert skips updates for existing rows. Must switch to `on_conflict_do_update` for `vote_count` and `vote_average` fields specifically, or vote_count will always be NULL for movies already in the DB.
- **Thumbnail JOIN in every endpoint:** Only `_build_session_response` needs thumbnails. Add a helper following the `_enrich_steps_watched_at` pattern ŌĆö keep all enrichment logic parallel and symmetric.
- **collapsible animation with CSS only:** Use Radix `Collapsible` (already available via shadcn primitives in the project) for accessible keyboard/ARIA behavior. A plain CSS height transition without Radix has no ARIA `aria-expanded`.

## Common Pitfalls

### Pitfall 1: `on_conflict_do_nothing` blocks vote_count updates

**What goes wrong:** `_ensure_actor_credits_in_db` uses `on_conflict_do_nothing` for movie rows. If a Movie was inserted before this phase (without `vote_count`), the upsert silently skips it. `vote_count` stays NULL forever.
**Why it happens:** `on_conflict_do_nothing` ignores the entire row if `tmdb_id` already exists.
**How to avoid:** Change to `on_conflict_do_update(set_={"vote_count": ..., "vote_average": ...})` for the fields being enriched.
**Warning signs:** `vote_count` is NULL for most movies in the DB after running the first eligibility check post-migration.

### Pitfall 2: MPAA fetch silent empty-string vs. None sentinel confusion

**What goes wrong:** `mpaa_rating=None` means "never fetched"; `mpaa_rating=""` means "fetched, no US cert found". If the migration sets all existing rows to `None`, every movie will be re-fetched once. After that, movies with no US cert have `""` and are skipped on re-fetch.
**Why it happens:** If you use `None` for both "never fetched" and "no rating found", every eligible-movies call tries to fetch MPAA for every movie.
**How to avoid:** Store `""` (empty string) as the "checked, no result" sentinel. In the filter UI, treat both `None` and `""` as "NR". In the re-fetch guard: `if movie.mpaa_rating is not None: skip`.

### Pitfall 3: runtime_map DB query N+1 for session counters

**What goes wrong:** Fetching `Movie.runtime` inside a loop over steps produces N queries.
**How to avoid:** Fetch all runtimes in a single `SELECT tmdb_id, runtime FROM movies WHERE tmdb_id IN (...)` ŌĆö same pattern as `_enrich_steps_watched_at`.

### Pitfall 4: Genre parsing from JSON string

**What goes wrong:** `Movie.genres` is stored as a JSON-encoded string e.g. `'["Action", "Drama"]'`. Parsing with `json.loads()` on a NULL value throws. Parsing `"null"` or `""` throws.
**How to avoid:** Always guard: `json.loads(m.genres) if m.genres else []`. Frontend: `const parseGenres = (s: string | null): string[] => { try { return JSON.parse(s ?? "[]") ?? [] } catch { return [] } }`.

### Pitfall 5: Filter sidebar collapses all genre checkboxes when list is long

**What goes wrong:** If a movie has 15+ distinct genres across the eligible list, the sidebar becomes very tall. User expected "collapsed by default".
**How to avoid:** Each filter category (Runtime, Genre, MPAA) has its own `Collapsible` ŌĆö collapsed by default. Expanding one does not expand others. This is already locked in the decisions.

### Pitfall 6: `include_ineligible` breaks existing `eligibleActors` query cache

**What goes wrong:** If the frontend passes `include_ineligible=true` but the existing `useQuery` key `["eligibleActors", sid, ...]` doesn't include the param, React Query may serve a cached response without the ineligible actors.
**How to avoid:** Include `include_ineligible` in the `queryKey`: `["eligibleActors", sid, session?.current_movie_tmdb_id, showIneligible]`.

### Pitfall 7: StepResponse thumbnail enrichment is called in ALL response-building paths

**What goes wrong:** `_build_session_response` is called in ~12 endpoints. Adding `_enrich_steps_thumbnails` to all of them adds DB queries to endpoints that don't need thumbnails (pause, resume, end, archive, etc.).
**How to avoid:** Only call `_enrich_steps_thumbnails` in the endpoints where thumbnails are actually displayed: `get_session_by_id` and `get_active_session`. Other endpoints (pause, resume, etc.) return `poster_path=None, profile_path=None` by default and that's acceptable.

    steps: list[GameSessionStep], db: AsyncSession
