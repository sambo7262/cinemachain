## Summary

Phase 4.3 is purely corrective and polish ŌĆö no new features, no new backend models or migrations. Every item has a clearly identified root cause discovered through direct source inspection. The work divides cleanly into: two backend-touching bugs (BUG-B session enrichment, BUG-C search), one backend prerequisite pull-in (04-08 short-circuit + nightly pre-fetch + stub backfill), four pure-frontend changes (BUG-A z-index/DOM, UX-A column sort, UX-B append pagination, UX-C stats trim, BUG-D layout), and one styling change (BUG-D button colors in GameLobby).

The most architecturally important finding: the `mark-current-watched` endpoint does NOT call `_enrich_steps_thumbnails` before building its response. It calls `_build_session_response` with only a `wa_map` ŌĆö no `poster_map` or `profile_map`. This is confirmed as the root cause of BUG-B: after mutation, all steps lose their `poster_path` because the response excludes thumbnail enrichment, and `setQueryData` writes that impoverished data into cache. The `get_session_by_id` endpoint does enrich thumbnails, and the 5-second polling refetch will eventually restore images ŌĆö but the gap causes visible blank-out.

For BUG-A, the PosterWall component renders at `z-[1]` and the content wrapper at `z-[2]`, which is correct. The `RadarrNotificationBanner` is already mounted outside PosterWall (it's in App.tsx above the route content div), but it is inside the `min-h-screen` container div that has no z-index, which means it is NOT above the PosterWall `z-[1]`. The real BUG-A issue is more likely path-based: `posterUrl()` constructs `/api${item.poster_local_path}` ŌĆö if `poster_local_path` is null and `poster_path` is also null, the `<img>` gets a URL like `https://image.tmdb.org/t/p/w185null` which produces a broken image.

**Primary recommendation:** Fix BUG-B first (highest impact: affects core game interaction every session), then BUG-A (affects every page load), then fold in 04-08 backend prerequisites before implementing BUG-C search, then UX changes as a batch, then BUG-D layout/styling last.

### BUG-A: PosterWall Image Rendering

**What's working:** PosterWall is correctly mounted. The blur overlay (`bg-black/80 z-10`) renders on top of poster columns. The component returns null when `posters.length < 5`.

**What's failing:** The `/movies/poster-wall` endpoint must be returning items where both `poster_local_path` and `poster_path` are problematic. In `posterUrl()`:
- If `poster_local_path` is set ŌåÆ uses `/api{poster_local_path}` (e.g. `/api/static/posters/550.jpg`) ŌĆö this path requires nginx to proxy `/api/static/` to backend StaticFiles mount.
- If `poster_local_path` is null ŌåÆ falls back to `https://image.tmdb.org/t/p/w185{poster_path}` ŌĆö this requires `poster_path` to be non-null and non-empty.

The NAS verification in 04.2-06 reported poster wall (5a) as FAIL, meaning the endpoint likely returns data but images don't render. Diagnosis path: (1) check if endpoint returns any movies with non-null poster paths, (2) check if nginx `/api/static/` proxy route exists, (3) verify StaticFiles is mounted at `/static/posters/` in the backend.

**BUG-A2 (Radarr banner blur):** The `RadarrNotificationBanner` in App.tsx renders OUTSIDE GameSession, so it is NOT inside PosterWall's stacking context (`z-[1]`). However, PosterWall uses `position: fixed; inset: 0`, which creates a new stacking context at `z-[1]`. The content wrapper inside GameSession at `z-[2]` overlaps PosterWall correctly. The notification banner in App.tsx at z-index auto is above both because it renders before the max-width content div in DOM order. **This means BUG-A2 may not be the issue originally feared ŌĆö the banner is already above PosterWall.** The real A2 risk would only manifest if the banner and PosterWall are in the same stacking context, which they are not. If blur IS affecting the banner, it may be a browser `filter` inheritance artifact on the outer container.

### BUG-B: Mark as Watched Image Blank-Out

**Confirmed root cause:** `mark_current_watched` endpoint at line 1034ŌĆō1079 builds its response with:
```python
wa_map = await _enrich_steps_watched_at(session.steps, db)
return _build_session_response(session, wa_map, current_movie_title=...)
```
No `poster_map`, no `profile_map`, no `runtime_map`. The `_build_session_response` call with `poster_map=None` produces `poster_path=None` for all steps. `markWatchedMutation.onSuccess` then calls `queryClient.setQueryData(["session", sid], updatedSession)` ŌĆö writing zero poster_paths into cache.

The session 5-second poller at `refetchInterval: 5000` would normally repair this, but the poller has `refetchInterval: (query) => query.state.data?.status === "awaiting_continue" ? false : 5000`. After `mark_current_watched`, status becomes `awaiting_continue` ŌĆö which **disables the poller entirely**. So the blank images are permanent until the user navigates away and back.

**Fix path 1 (data):** Add `_enrich_steps_thumbnails` to `mark_current_watched` endpoint. This is the right fix ŌĆö the data should be complete.

**Fix path 2 (UI resilience):** Add `placeholderData: keepPreviousData` to the session query so stale-but-correct data shows during any refetch gap. This is the CONTEXT.md-specified fix and complements fix path 1.

Both fixes should be applied: data fix in backend, UI resilience fix in frontend.

### UX-A: Column Sort ŌĆö Current State vs Target State

**Current state (confirmed):** A `<Select>` dropdown with three options (By Rating, By Runtime, By Genre) controls the `sort` state variable. Sort values are `"rating" | "runtime" | "genre"`. The sort state is passed via `api.getEligibleMovies(...)` ŌåÆ backend `sort` query param. Backend sorts server-side before pagination.

**Target state:** Remove the `<Select>` component. Add click handlers on `<th>` elements for Rating, Year, Runtime, MPAA columns. State becomes `{ column: "rating" | "year" | "runtime" | "mpaa", direction: "asc" | "desc" }`. Default: `{ column: "rating", direction: "desc" }`.

**Backend impact:** The backend currently accepts `sort = "rating" | "runtime" | "genre"`. Year sort does not exist. MPAA sort does not exist. The backend sort param must be extended to support `year_asc`, `year_desc`, `rating_asc`, `rating_desc`, `runtime_asc`, `runtime_desc`, `mpaa_asc`, `mpaa_desc` ŌĆö OR the frontend can pass direction as a separate query param. The simplest extension: add `year` and `mpaa` as sort options on the backend, add a `sort_dir` query param (`asc` | `desc`). Alternatively, prefix-based: `rating_desc`, `year_asc`, etc.

**Key consideration:** Genre sort is currently an option ŌĆö it is NOT one of the four columns getting clickable headers (Rating, Year, Runtime, MPAA). Genre is handled by the sidebar filter (not a sort column). The genre sort dropdown option can be dropped since filters cover it.

### UX-B: Load More Append ŌĆö Current State vs Target State

**Current state (confirmed):** `moviesPage` state increments on Load More click. But `queryKey: ["eligibleMovies", sid, selectedActor?.tmdb_id ?? null, sort, allMovies, moviesPage]` ŌĆö each page is a separate cache entry. The query returns only page N's items, replacing the displayed list entirely (since `allEligibleMovies = eligibleMoviesData?.items ?? []`).

**Target state:** Maintain a `accumulatedMovies` state array (or equivalent) that persists across page increments. On new page fetch, append items rather than replace. The `moviesPage` in the queryKey already distinguishes pages ŌĆö the fix is to accumulate results client-side.

**Scroll-to-new pattern:** Use a `ref` attached to the first newly appended item via a `data-first-new` attribute or by tracking the pre-append count. After state update settles, call `scrollIntoView`.

**Reset on actor/filter/search change:** When `selectedActor`, `sort`, `allMovies`, or search changes, reset accumulated list to the new first page.

**Load More hidden when exhausted:** `eligibleMoviesHasMore` already derives from `has_more` in the response. When false AND search/filter is active, hide the button.

### BUG-C: Backend Search Implementation

**Current state:** Search is entirely client-side. `filteredMovies` filters `allEligibleMovies` by `m.title.toLowerCase().includes(debouncedSearch.toLowerCase())`. The backend `get_eligible_movies` endpoint has no `search` parameter.

**Target state:** Add `search: str | None = Query(default=None)` to `get_eligible_movies`. When `search` is provided:
1. Call `_ensure_actor_credits_in_db` for the selected actor (if actor_id is not None) to guarantee full filmography coverage.
2. Filter `movies` list by title match (case-insensitive `search` in `movie["title"]`) before pagination.
3. Return ALL matching results with `page=1, page_size=len(movies), has_more=False` (effectively disabling pagination for search results).

**Frontend changes:**
- Pass `search: debouncedSearch || undefined` to `api.getEligibleMovies` when search is active.
- Add `debouncedSearch` to the queryKey.
- Remove the client-side `debouncedSearch` filter from `filteredMovies` (since backend now handles it).
- The backend handles search as a subset of all results (no pagination), so `eligibleMoviesHasMore` will be false ŌĆö Load More button auto-hides.
- When search is cleared, the query reverts to paginated mode.

**api.ts change:** Add `search?: string` to `getEligibleMovies` params and include `if (params?.search) q.set("search", params.search)`.

## Common Pitfalls

### Pitfall 1: BUG-B ŌĆö Fixing only the frontend without fixing the backend

**What goes wrong:** Adding `placeholderData: keepPreviousData` hides the blank image problem during the brief refetch window. But the mutation `onSuccess` calls `queryClient.setQueryData` with the impoverished response (no poster_paths) ŌĆö overwriting the cached good data immediately. `keepPreviousData` only protects during the async query refetch, not during synchronous `setQueryData` calls.

**How to avoid:** Fix BOTH: (1) add thumbnail enrichment to `mark_current_watched` backend endpoint, AND (2) add `placeholderData: keepPreviousData` to the session query for resilience.

### Pitfall 2: UX-B ŌĆö Accumulated movies not resetting on filter/actor/sort change

**What goes wrong:** User selects actor A ŌåÆ sees page 1 ŌåÆ clicks Load More ŌåÆ sees pages 1+2. User then selects actor B. If accumulated state is not cleared, they see actor A's page 2 results alongside actor B's page 1.

**How to avoid:** `useEffect` that watches `[selectedActor?.tmdb_id, sort, allMovies]` and resets accumulated state and `moviesPage` to 1 when any of these change.

### Pitfall 3: BUG-C ŌĆö Search bypassing client-side filters

**What goes wrong:** When search is active, backend handles filtering. But the client-side `filteredMovies` still applies genre/runtime/MPAA filters on top. If the genre sidebar remains active while search is active, results may be unexpectedly narrow.

**How to avoid:** When search is active (`debouncedSearch !== ""`), pass search to backend AND skip client-side search filtering. Sidebar filters (genre, runtime, MPAA) should continue to apply client-side on top of search results ŌĆö this is correct behavior and matches the CONTEXT.md spec ("same behavior as filters"). The key fix: remove the client-side search filter from `filteredMovies` when search is active (since backend now handles it).

### Pitfall 4: UX-A ŌĆö Backend sort param mismatch

**What goes wrong:** Frontend sends `sort=year_desc` but backend only accepts `"rating" | "runtime" | "genre"`. Backend receives unknown sort value and applies no sort (or falls through without sorting).

**How to avoid:** Add `year` and `mpaa` as sort options to the backend `get_eligible_movies` endpoint's sort logic, and add `sort_dir: str = Query(default="desc")` or use combined `sort=year&sort_dir=desc` params. Ensure the frontend and backend agree on the sort param encoding before implementing either side.

### Pitfall 5: UX-B ŌĆö queryKey invalidation breaking accumulated state

**What goes wrong:** When `moviesPage` increments, the new queryKey differs from the previous one. If `queryClient.invalidateQueries` is called on `["eligibleMovies", sid]` (e.g. from markWatched mutation), React Query refetches page 1 with fresh data ŌĆö but the accumulated state still shows pages 1+2.

**How to avoid:** On mutation success that invalidates `eligibleMovies`, also reset `moviesPage` to 1 and clear the accumulated list. The `markWatchedMutation.onSuccess` already invalidates `eligibleMovies` ŌĆö add the reset there.

### Pitfall 6: BUG-A ŌĆö z-index stacking with `fixed` PosterWall

**What goes wrong:** PosterWall is `position: fixed; z-[1]`. Content wrapper is `position: relative; z-[2]`. If any ancestor of the content wrapper has a CSS filter, transform, or opacity less than 1, it creates a new stacking context that may cause the `z-[2]` to be scoped within that context ŌĆö making it effectively below `z-[1]` in the global stack.

**How to avoid:** Ensure no ancestor of the content wrapper `div.relative.z-[2]` applies filter, transform, or opacity. The `blur-[20px]` on PosterWall column containers is scoped to the PosterWall children, not ancestors of the content wrapper.

    page_size: int = Query(default=20, ge=1, le=100),
    db: AsyncSession = Depends(get_db),
):

    # After filtering and sorting, apply search:
    if search:
        search_lower = search.lower()
        movies = [m for m in movies if search_lower in m["title"].lower()]
        # Return all results (no pagination) for search
        total = len(movies)
        return {
            "items": movies,
            "total": total,
            "page": 1,
            "page_size": total,
            "has_more": False,
        }

    # Normal pagination path (unchanged):
    total = len(movies)
    offset = (page - 1) * page_size
    movies = movies[offset : offset + page_size]
    return {
        "items": movies,
        "total": total,
        "page": page,
        "page_size": page_size,
        "has_more": (offset + page_size) < total,
    }
