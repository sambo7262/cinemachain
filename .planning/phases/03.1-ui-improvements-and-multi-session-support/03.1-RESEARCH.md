Phase 03.1 is a pure improvement phase — no game logic changes. All eight GAME requirements are already satisfied. The work falls into four tracks:

**Track 1 (Backend):** Add `name` and `archived` fields to `GameSession`, remove the single-session enforcement gate, add a `GET /game/sessions` list endpoint, add a `POST /game/sessions/{id}/archive` endpoint, and add `GET /game/sessions/archived`. **Primary recommendation:** Implement migration 0004 first (name + archived columns), then backend list/archive endpoints, then frontend in dependency order: api.ts → Home page → Session page → New routes.

### Pattern 1: Multi-Session Backend Gate Removal

**What:** Two locations enforce the single-session constraint:
1. `_get_single_active_session()` helper — used by `create_session` and `import_csv_session`
2. Both endpoints raise HTTP 409 if `existing` is not None

**How to remove:**
- Delete the 409 checks from `create_session` and `import_csv_session`
- Keep `_get_single_active_session` (or rename to `_get_active_session_by_id`) for internal use elsewhere, but stop using it as a gate
- `GET /game/sessions/active` currently returns a single session — replace with `GET /game/sessions` returning a list of active sessions

**Critical:** `get_active_session` is called by `NavBar.tsx` and `GameLobby.tsx` via `queryKey: ["activeSession"]`. This query must be updated to use the new list endpoint, and all callers must handle an array instead of a single object.

### Pattern 2: Session Name Uniqueness

**Server-side enforcement approach:**

Option A (recommended): Query-check before insert
```python
# In create_session and import_csv_session:
existing_name = await db.execute(
    select(GameSession)
    .where(GameSession.name == body.name)
    .where(GameSession.status.not_in(["archived", "ended"]))
)
if existing_name.scalar_one_or_none():
    raise HTTPException(status_code=409, detail="Session name already in use")
```

Option B: DB UniqueConstraint filtered by status — PostgreSQL supports partial unique indexes:
```sql
CREATE UNIQUE INDEX uq_game_sessions_name_active
ON game_sessions (name)
WHERE status NOT IN ('archived', 'ended');
```

Option B is cleaner and race-condition-safe. Alembic supports raw SQL index creation via `op.execute()`. Recommended.

**Client-side enforcement:** Before submitting, query active sessions and check for name collision in JS. The server-side partial index is the authoritative guard; client-side is UX-only.

### Pattern 3: Archived Status Column

**Recommendation:** Add `archived` as a new `SessionStatus` enum value (`"archived"`) rather than a separate boolean column. Reason: `status` is already a `String(20)` (not PostgreSQL ENUM — see STATE.md), so adding `"archived"` requires only:

1. Migration: `op.execute("UPDATE ... SET status='archived' WHERE ...")` — no schema change needed for the string column itself
2. ORM: Add `archived = "archived"` to `SessionStatus` enum
3. Alembic migration 0004: add `name` column + the partial unique index

Migration 0004 schema:
```sql
ALTER TABLE game_sessions ADD COLUMN name VARCHAR(100) NOT NULL DEFAULT '';
ALTER TABLE game_sessions ADD COLUMN archived_at TIMESTAMP NULL;
CREATE UNIQUE INDEX uq_game_sessions_name_active
  ON game_sessions (name)
  WHERE status NOT IN ('archived', 'ended');
```

**Note on `name` NOT NULL with DEFAULT '':** Existing rows in production get an empty string name. The plan must handle backfilling names for existing sessions (or accept empty names for pre-existing sessions, which will never surface in the UI since they are already ended or will be archived on next access).

### Pattern 4: New Backend Endpoints

```
GET  /game/sessions                     → list[GameSessionResponse]  (active only, default)
GET  /game/sessions?include_archived=true → list[GameSessionResponse]  (all)

This is the same pattern applied to `import-csv` in the existing code (see STATE.md).

### Pattern 5: Chain History Table Redesign

The current `ChainHistory` is a horizontal scrollable flow of nodes. The new design is a vertical numbered table with alternating Movie and Actor rows.

**Step data availability:** `GameSessionStepDTO` already carries `movie_title`, `actor_name`, `movie_tmdb_id`, `actor_tmdb_id`. What it does NOT carry is a watched timestamp. The `watched_at` for a movie in the chain corresponds to the `WatchEvent.watched_at` for that movie's `tmdb_id`.

**Two options for timestamps:**
- Option A (recommended): Backend enriches `StepResponse` / `GameSessionResponse` with `watched_at` by joining `WatchEvent` on `movie_tmdb_id`. Adds a single LEFT JOIN to the session fetch.
- Option B: Frontend makes a separate `/movies/watched` call and resolves timestamps client-side. Works but is a second round-trip and fragile coupling.

Use Option A. Add `watched_at: datetime | None` to `StepResponse` and `GameSessionStepDTO`.

**Image sizing:**
- Movie poster thumbnail: `https://image.tmdb.org/t/p/w92{poster_path}` — already used in MovieCard
- Actor profile thumbnail: `https://image.tmdb.org/t/p/w92{profile_path}` — already used in GameSession eligible-actors table

**Row structure per step:**

For step with `actor_name` present (non-starting steps):
- Row N (Movie): `#N | poster | movie_name | watched_at timestamp`
- Row N (Actor): `#N | profile_photo | actor_name`

For step_order=0 (starting movie, actor_tmdb_id is NULL):
- Row 0 (Movie only): `#0 | poster | movie_name | watched_at timestamp`

### Pattern 6: CSV Export

Backend endpoint returns `StreamingResponse` with `Content-Disposition: attachment; filename="chain.csv"`. Format matches import format exactly for roundtrip.

Format (interleaved — recommended for readability and clean roundtrip):
```csv
order,movie_name,actor_name
0,Free Guy,
0,,Ryan Reynolds
1,Deadpool and Wolverine,
```

OR flat pairs: one row per movie with actor in same row (simpler, but breaks roundtrip symmetry). Use interleaved to match import schema — the import parser already handles rows with blank movie_name or blank actor_name.

The export endpoint reads `session.steps` and emits one or two rows per step based on whether `actor_name` is present.

### Pattern 7: CSV Import Validation Step

Current flow: file parsed → immediate import button.

New flow: file parsed → validation/preview step → user reviews/edits → import.

Validation rules:
1. Strip all columns except `order`, `movie_name`, `actor_name` (client-side before preview)
2. Sequence check: a valid chain must alternate movie → actor → movie → actor (or start with movie only)
   - A row with non-empty `movie_name` and empty `actor_name` = movie row
   - A row with empty `movie_name` and non-empty `actor_name` = actor row
   - Two consecutive movie rows without an actor row = invalid
   - Two consecutive actor rows = invalid
3. Rows with both empty `movie_name` and `actor_name` = filtered out silently

UI: Display parsed rows as a list. Invalid rows highlighted (red border or warning icon). Each row has a delete button (×). "Import Chain" button disabled if sequence is invalid. Optional: "Force import anyway" for power users.

**Column stripping in parser:**

```typescript
// New parseCSV: detect column positions by header names, not position
function parseCSV(text: string): ParsedRow[] {
  const lines = text.trim().split("\n")
  const headers = lines[0].split(",").map(h => h.trim().toLowerCase().replace(/^"|"$/g, ""))
  const orderIdx = headers.indexOf("order")
  const movieIdx = headers.findIndex(h => h.includes("movie"))
  const actorIdx = headers.findIndex(h => h.includes("actor"))
  return lines.slice(1).map(line => {
    const cols = line.split(",").map(c => c.trim().replace(/^"|"$/g, ""))
    return {
      order: parseInt(cols[orderIdx] ?? "0") || 0,
      movieName: cols[movieIdx] ?? "",
      actorName: cols[actorIdx] ?? "",
    }
  }).filter(r => r.movieName || r.actorName)
}
```

### Pattern 8: Home Page — Session Grid

`GameLobby.tsx` currently shows either "session in progress" banner OR start panel. With multi-session support it becomes:

```
Header: CinemaChain / Navigate cinema through shared actors

[Active Sessions grid]
  Card 1: Session Name | Current Movie | Chain length | [Continue →]
  Card 2: ...
  [+ Start a new session] card/button

[If no sessions: "No active sessions. Start one to begin."]
```

The "Start a new session" action opens an inline form (or expands a section) with:
- Name input (required, validated for uniqueness against active sessions)
- Three tabs: Watch History | Search Title | Import Chain (same as current)

The home page query changes from `getActiveSession()` to a new `listActiveSessions()` call returning `GameSessionDTO[]`.

### Anti-Patterns to Avoid

- **Calling `getActiveSession` from NavBar for multi-session world:** The NavBar currently polls `getActiveSession` to determine the Sessions link href. With multiple sessions, this becomes ambiguous. The NavBar "Sessions" link should simply route to `"/"` always. No per-session routing from NavBar needed.
- **Sharing queryKey `["activeSession"]` for the list endpoint:** Rename to `["activeSessions"]` (plural) to avoid cache collisions with any remaining single-session calls.
- **Blocking import on single-session gate:** The `import_csv_session` endpoint currently raises 409 if an active session exists. This gate must be removed, same as `create_session`.
- **Trusting `movie_title` being populated:** Several steps were created before `movie_title` was stored (early game sessions). Always provide a fallback: `step.movie_title ?? "(untitled)"` — never surface TMDB IDs.
- **read-only enforcement via frontend only:** Archived sessions must be guarded server-side too. Add a status check to `pick-actor`, `request-movie`, `mark-current-watched`, and `continue-chain` endpoints — return 422 if session is archived.

## Common Pitfalls

### Pitfall 1: Route Registration Order for Static Paths

**What goes wrong:** FastAPI tries to parse `"archived"` or `"export-csv"` as an integer `session_id` path param, returns 422 Unprocessable Entity.

**Why it happens:** `/{session_id}` is registered before `/archived`.

**How to avoid:** Register `/sessions/archived` and `/sessions/{id}/export-csv` (static segment) before `/{session_id}` catch-all. This is documented in STATE.md for `import-csv` and already applied.

**Warning signs:** 422 error with `"value is not a valid integer"` for the path param.

### Pitfall 2: Query Cache Key Collision

**What goes wrong:** NavBar and GameLobby both use `queryKey: ["activeSession"]` pointing at `api.getActiveSession()`. Changing `getActiveSession` to return a list or a different shape silently breaks all callers.

**How to avoid:** Introduce a new `queryKey: ["activeSessions"]` for the list endpoint. Keep `["activeSession"]` only if a single-session lookup is still needed anywhere (it's not for this phase). Grep for all `["activeSession"]` usages before changing the API shape.

**Files affected:** `GameLobby.tsx` (line 20), `NavBar.tsx` (line 9).

### Pitfall 3: `watched_at` Timestamp Not in StepResponse

**What goes wrong:** Chain history table needs to display when each movie was watched. `GameSessionStep` does not store `watched_at`. `WatchEvent` does, keyed on `tmdb_id`.

**How to avoid:** Add `watched_at: datetime | None` to `StepResponse` by LEFT JOINing `WatchEvent` in the session-fetch query (or enriching after fetch). Do this in the backend before writing the frontend table — the frontend cannot display data that the API doesn't return.

### Pitfall 4: `movie_title` NULL for Old Steps

**What goes wrong:** `GameSessionStep.movie_title` is nullable (String(255), nullable=True). Steps created before `movie_title` was reliably stored may have NULL. The current code already has the fallback `` `Movie ${session.current_movie_tmdb_id}` `` in two places. The new chain history table will expose this bug at higher visibility (every row).

**How to avoid:** Never render TMDB IDs in user-facing text. Always apply fallback: `step.movie_title ?? "(untitled)"`. Backend enrichment is optional — the title is available in the `Movie` table via `tmdb_id`; the backend could backfill during a migration.

### Pitfall 5: Archived Session Read-Only Enforcement

**What goes wrong:** Frontend shows archived sessions as read-only, but backend endpoints (`pick-actor`, `request-movie`, etc.) don't check archived status. A user navigating directly to `/game/{id}` for an archived session could still make API calls.

**How to avoid:** Add `if session.status == "archived": raise HTTPException(422, "Session is archived")` at the start of all mutating session endpoints.

### Pitfall 6: Import CSV Partial Unique Index Race Condition

**What goes wrong:** Two users submit a session with the same name simultaneously; application-level check passes for both; both rows insert; one fails at DB with `IntegrityError`.

**How to avoid:** The partial unique index handles this at the DB level. Catch `IntegrityError` from SQLAlchemy and return 409.

### Pitfall 7: `name` Column NOT NULL Default on Existing Rows

**What goes wrong:** Migration adds `name VARCHAR(100) NOT NULL` without a server default. PostgreSQL rejects the migration because existing rows have no value.

**How to avoid:** Always provide `server_default=''` (empty string) in the migration, or backfill existing rows first. The recommended migration uses `server_default=''`.

### Pitfall 8: CSV Export Column Order Must Match Import Parser

**What goes wrong:** Export produces `movie_name,actor_name,order`; import parser assumes `order` is in position 2 (current code: `const [movieName, actorName, order] = line.split(",")`). Roundtrip breaks.

**How to avoid:** The new import parser uses header-name lookup (Pitfall 4 in Don't Hand-Roll). The export should produce `order,movie_name,actor_name` (matching CONTEXT.md spec). The new import parser detects column positions by header name, so any column order works.

async def list_sessions(
    include_archived: bool = Query(default=False),
    db: AsyncSession = Depends(get_db),
):
    stmt = select(GameSession).options(selectinload(GameSession.steps))
    if include_archived:

        stmt = stmt.where(GameSession.status.not_in(["ended", "archived"]))

    return [GameSessionResponse.model_validate(s) for s in sessions]

### Archive endpoint

```python
@router.post("/sessions/{session_id}/archive", response_model=GameSessionResponse)
async def archive_session(session_id: int, db: AsyncSession = Depends(get_db)):
    from datetime import datetime as _datetime
    result = await db.execute(
        select(GameSession).where(GameSession.id == session_id)
        .options(selectinload(GameSession.steps))
    )
    session = result.scalar_one_or_none()
    if not session:
        raise HTTPException(status_code=404, detail="Session not found")
    session.status = "archived"
    session.archived_at = _datetime.utcnow()
    await db.commit()
    result = await db.execute(
        select(GameSession).where(GameSession.id == session.id)
        .options(selectinload(GameSession.steps))
    )
    session = result.scalar_one()
    return GameSessionResponse.model_validate(session)
```

### CSV export endpoint

```python
from fastapi.responses import StreamingResponse
import io, csv

@router.get("/sessions/{session_id}/export-csv")
async def export_csv(session_id: int, db: AsyncSession = Depends(get_db)):
    result = await db.execute(
        select(GameSession).where(GameSession.id == session_id)
        .options(selectinload(GameSession.steps))
    )
    session = result.scalar_one_or_none()
    if not session:
        raise HTTPException(status_code=404, detail="Session not found")

    output = io.StringIO()
    writer = csv.writer(output)
    writer.writerow(["order", "movie_name", "actor_name"])
    for step in sorted(session.steps, key=lambda s: s.step_order):
        writer.writerow([step.step_order, step.movie_title or "", ""])
        if step.actor_name:
            writer.writerow([step.step_order, "", step.actor_name])

    output.seek(0)
    filename = f"chain-{session.name or session.id}.csv"
    return StreamingResponse(
        iter([output.getvalue()]),
        media_type="text/csv",
        headers={"Content-Disposition": f'attachment; filename="{filename}"'},
    )
```

## Open Questions

1. **`GET /game/sessions/active` — keep or replace?**
   - What we know: NavBar and GameLobby call `api.getActiveSession()` → `GET /game/sessions/active`
   - What's unclear: Is there a clean way to keep this endpoint for backwards compat while also adding the list endpoint?
   - Recommendation: Replace `getActiveSession` with `listSessions` in NavBar and GameLobby. Remove the old endpoint or leave it returning `null` always (to avoid 404 surprises). The simplest path: rename the endpoint to `GET /game/sessions` and have NavBar route to `/` without session-specific href.

2. **`watched_at` enrichment — migration or query-time?**
   - What we know: `WatchEvent` has `watched_at` keyed on `tmdb_id`. `GameSessionStep` has `movie_tmdb_id`.
   - What's unclear: Whether to store `watched_at` on the step (denormalized) or join at query time
   - Recommendation: Join at query time (single LEFT JOIN). Avoids data duplication. Add to `StepResponse` as `watched_at: datetime | None`.

3. **Name input placement on home page**
   - What we know: Session name is required at creation time
   - What's unclear: Whether name input appears inline on the home page or in a modal/drawer
   - Recommendation: Inline expansion — clicking "+ Start a new session" expands a form section with name input + the three existing tabs. No modal needed; matches current app's no-modal philosophy (confirmed: `window.confirm` used elsewhere).

4. **Backfilling names for existing active sessions in production**
   - What we know: Existing active session(s) will get `name = ""` after migration 0004
   - What's unclear: Whether to auto-generate names (e.g., "Session 1") or leave blank
   - Recommendation: Migration 0004 uses `server_default=''`. The plan should include a migration step that sets `name = 'Session ' || id` for existing active sessions via `op.execute("UPDATE game_sessions SET name = 'Session ' || id::text WHERE name = ''")`.
