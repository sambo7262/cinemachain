## Root Cause Analysis: BUG-03 Remaining Failure

The original diagnosis in RESEARCH.md correctly identified the `if not eligible_actors: return []` early return as blocking the fallback. The fix was applied correctly. However, the diagnosis was **incomplete** — it did not account for the second condition gating the fallback.

**The full call path when eligible_actors is empty:**

1. `eligible_actors = []` (all cast have been picked)
2. `candidates = []` (guard correctly skips candidate query) — FIX APPLIED, CORRECT
3. `best = {}`, `top5 = []` (scoring loop over empty candidates produces nothing)
4. Fallback gate: `if len(top5) < 5 and genre_freq:` → `if True and genre_freq:`
   - If `genre_freq` is non-empty (watch history exists): fallback block executes
   - If `genre_freq` is `{}` (no watch history): `and {}` evaluates to `False` — **fallback skipped, returns `[]`**

**Why genre_freq may be empty on a real 8-step session:**

`genre_freq` is built from two sources:
1. WatchEvents joined to Movie (line 1706-1717) — requires rows in the `watch_events` table for chain movies
2. Session step movies joined to Movie.genres (line 1720-1731) — this ALWAYS has data (8 chain movies)

Source 2 is unconditional and should always produce genre data from the session's own chain movies. However, the query at line 1720 selects from `Movie.genres` where `Movie.tmdb_id.in_(picked_movie_ids)`. If the chain movies were imported via CSV but their `genres` field in the Movie table is NULL (not yet enriched from TMDB), this source also returns nothing.

**Conclusion:** The most likely failure scenario for an 8-step chain is that the chain movies' `Movie.genres` field is NULL (TMDB enrichment not yet complete or the movie stub was created without genre data), causing both genre_freq sources to yield empty results, making the fallback gate condition false.

**Secondary contributor:** Even when the fallback block executes, `if genre_score > 0:` at line 1783 excludes any candidate whose genres do not overlap with genre_freq. With a thin or generic genre_freq, this filter may produce an empty fallback_scored list, still returning `[]`.

## Gaps Summary

One gap blocks phase goal achievement:

**BUG-03 is not resolved.** The code fix (removing the early return in `get_suggestions`) is correctly applied, but the diagnosis was incomplete. The genre-affinity fallback at line 1764 has a second gating condition — `and genre_freq` — that causes the function to return `[]` whenever watch history is absent or chain movie genres are not populated in the DB. For an 8-step test session, if the chain movies do not have `genres` data populated in the Movie table, `genre_freq` will be an empty dict, the fallback block will be skipped entirely, and the endpoint returns `[]`.

The fix requires:
1. Making the fallback unconditional when `top5` is empty (remove `and genre_freq` as a total-gate; move genre scoring inside the block as a ranking tool rather than an enable condition)
2. Removing or softening `if genre_score > 0:` so that candidates with zero genre overlap are still returned (ranked last) when no better options exist
3. Updating the regression test to include a variant with no WatchEvents so the zero-watch-history path is covered

The regression test (`test_suggestions_long_chain_fallback`) seeds a WatchEvent in its setup, which masks this failure path. The test will pass in Docker even though the real failure scenario is not covered.
