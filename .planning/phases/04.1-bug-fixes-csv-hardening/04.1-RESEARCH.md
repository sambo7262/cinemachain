All three have been partially addressed in plans 04-08 and 04-09 (code changes to `game.py` and `cache.py`), but those plans were written after Phase 4's verification checkpoint (04-07) and were never deployed to the NAS. The primary work in this phase is therefore: (1) fix BUG-02 which was NOT addressed by 04-09, (2) verify the 04-08/04-09 fixes for BUG-01 and BUG-03 actually resolve the reported issues, and (3) deploy and verify on the live NAS.

**Current state of each bug:**
- **BUG-01** (CSV chain movie eligibility): Fix exists in `game.py` lines 1283-1395. Chain movie exclusion uses `chain_movie_ids = set(session_movie_ids)` and filters `[m for m in movies if m["tmdb_id"] not in chain_movie_ids]`. Logic looks correct but has never been deployed since the pre-04-07 deploy. Needs NAS verification.
- **BUG-02** (CSV comma-in-title parsing): NOT fixed. Frontend `parseCSV()` in `GameLobby.tsx` lines 34-34 uses `line.split(",")` — a naive split that breaks when movie titles contain commas. Fix requires proper quoted-CSV parsing (RFC 4180 compliance) in the frontend. No backend change needed for this approach; the `.xlsx` alternative also remains viable but adds `openpyxl` dependency.
- **BUG-03** (Suggested movies empty on long chains): Fallback exists in `game.py` lines 1769-1798. The root cause is different from what was assumed — the fallback triggers when `eligible_actors` is empty (line 1686 returns `[]` early, before the fallback code at 1769 is reached). The fallback only runs when candidates exist but `len(top5) < 5`. This means the fallback does NOT help when the current movie has no eligible actors in DB (long chain exhausts all actors). Deploy is still needed to verify whether the fallback addresses the actual long-chain failure mode.

**Primary recommendation:** Fix BUG-02 with RFC 4180-compliant CSV parsing in the frontend (the `parseCSV` function). Deploy all three changes together in a single Docker rebuild + NAS verify plan.

### BUG-02 Option A: Fix frontend CSV parser (RECOMMENDED)
No new dependencies. The frontend `parseCSV` function replaces `line.split(",")` with proper quoted-field handling or delegates to a well-known pattern.

**Pattern — RFC 4180-compliant split in TypeScript:**
The standard approach is to use a regex that handles quoted fields:
```typescript
// Splits a CSV line respecting quoted fields containing commas
function splitCsvLine(line: string): string[] {
  const result: string[] = []
  let current = ""
  let inQuotes = false
  for (let i = 0; i < line.length; i++) {
    const ch = line[i]
    if (ch === '"') {
      if (inQuotes && line[i + 1] === '"') { current += '"'; i++ }
      else { inQuotes = !inQuotes }
    } else if (ch === "," && !inQuotes) {
      result.push(current.trim()); current = ""
    } else {
      current += ch
    }
  }
  result.push(current.trim())
  return result
}
```
Confidence: HIGH — standard RFC 4180 parsing logic, no external dependency.

### BUG-02 Option B: Add .xlsx support (ALTERNATIVE — more complex)
| Library | Version | Purpose | Tradeoff |
|---------|---------|---------|----------|
| openpyxl | >=3.1.0 | Parse .xlsx files in Python backend | Adds ~3MB to image, requires backend endpoint change or format detection |

Not recommended unless users specifically need Excel format. The CSV export from the app itself produces well-formed quoted CSV (Python's `csv.writer` quotes fields correctly), so the round-trip case already works. The comma-in-title failure only affects externally-authored CSV files.

**Recommendation:** Option A (fix frontend parser). UI-SPEC already documents this decision at "Option B (additive)" but the simplest first fix is the frontend-only parser fix with zero new dependencies.

## Architecture Patterns

### BUG-01: Chain Movie Eligibility

**Root cause confirmed:** The fix is already in `game.py`. `chain_movie_ids` is built from `session.steps` at line 1284-1285, and the filter at line 1395 removes all chain movies from the result. The `selectable` field also accounts for chain movies at lines 1324 and 1364.

**Potential remaining issue:** The `watched_ids` query at lines 1289-1294 scopes WatchEvents to `session_movie_ids` (i.e., only events for movies in this session's chain). This is correct per the 03.2-15 decision. However, on a CSV-imported chain, WatchEvents for prior movies are created with `source="csv_import"` (lines 720-728). These should be returned by the query and mark those movies as `watched=True`. The `selectable` field then becomes `False` for both watched and chain movies — correct behavior.

**Why "not resolving on NAS":** Plans 04-08 and 04-09 were written as deferred post-04-07 plans. The 04-07 checkpoint only verified plans 04-01 through 04-06. Plans 04-08 and 04-09 code changes exist in the local repo but have never been built into Docker images or deployed to the NAS.

**Verification requirement:** After deploy, test with a CSV-imported session — chain movies must not appear in the Eligible Movies tab.

### BUG-02: CSV Comma-in-Title Parsing

**Root cause:** `GameLobby.tsx` `parseCSV()` function at line 34 uses `line.split(",")`. When a movie title contains a comma (e.g., `"Beauty and the Beast, The"`), the split produces extra columns, shifting `movieName` and `actorName` to wrong column indices.

**Fix location:** Frontend only — `parseCSV` function, lines 24-44 of `GameLobby.tsx`. Replace `line.split(",")` with an RFC 4180-aware split. The header line at line 26 also uses `split(",")` and must be fixed for consistency, though headers rarely contain commas.

**Python's csv.writer already quotes correctly:** The Export CSV function at game.py line 917 uses `csv.writer(output)` which RFC 4180-quotes fields containing commas automatically. So any CSV exported from the app and re-imported will work IF the frontend parser handles quotes correctly.

**No backend change needed** unless Option B (.xlsx) is chosen.

### BUG-03: Suggested Movies Empty on Long Chains

**Root cause analysis (from code):** The suggestions endpoint at line 1686 returns `[]` immediately when `eligible_actors` is empty:
```python
if not eligible_actors:
    return []
```
On a long chain (130+ movies), ALL actors from the current movie have been picked in prior steps. `get_eligible_actors` with `Actor.tmdb_id.notin_(picked_actor_ids)` returns zero rows. The fallback at lines 1769-1798 is never reached because execution exits at line 1687.

**The fallback only runs when `eligible_actors` is non-empty but `len(top5) < 5`.** It does NOT address the zero-eligible-actors case.

**Fix required:** Move the fallback logic to also cover the case where `eligible_actors` is empty. Specifically, when `eligible_actors` returns empty, skip straight to genre-affinity scoring from the full DB (the existing fallback code) rather than returning `[]`.

**Revised algorithm:**
1. Get eligible actors (cast of current movie, excluding picked actors)
2. If eligible actors exist: compute game-mechanic candidates (existing steps 3-6)
3. If top5 < 5 OR eligible actors is empty: fill/replace with genre-affinity fallback from full DB
4. Return top5 (up to 5 results, may be all genre-affinity fallback on long chains)

This is a small surgical change — add an `or not eligible_actors` condition to the existing fallback guard at line 1769, and reorder so the fallback path is reachable even when the early return at 1686 would have fired.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| RFC 4180 CSV parsing | Custom regex | Standard quoted-field state machine | Edge cases: escaped quotes `""`, CRLF inside fields, BOM characters |
| Quoted CSV header parsing | Separate header logic | Same `splitCsvLine` function | Consistency — header row may also contain quoted fields |

**Key insight:** The Python `csv` module (already imported at game.py line 4) correctly handles all RFC 4180 edge cases. The frontend is the only broken parser; it does not need to match all edge cases of the Python module, but it must handle `"field,with,commas"` and `""escaped quotes""`.

## Common Pitfalls

### Pitfall 1: Early Return Masking Fallback
**What goes wrong:** BUG-03 fallback at line 1769 is unreachable when `eligible_actors` is empty. The `if not eligible_actors: return []` guard at line 1686 exits before the fallback.
**Why it happens:** The fallback was added as a `len(top5) < 5` supplement, not as a zero-actors rescue.
**How to avoid:** The fix must restructure the function to either (a) not early-return on empty eligible_actors, or (b) call the fallback explicitly from the early-return branch.
**Warning signs:** `return []` anywhere before the fallback code block.

### Pitfall 2: CSV Chain Movie Exclusion — Subtle Watched/Selectable Split
**What goes wrong:** On a CSV import, chain movies appear as "watched" (WatchEvent exists from csv_import) but `selectable=False` — both because they're in `watched_ids` AND in `chain_movie_ids`. The filter at line 1395 removes them entirely so they don't appear at all. But if the exclusion filter is applied AFTER pagination, chain movies could appear on earlier pages with `selectable=False` visible to the user.
**Current code:** Line 1395 filters BEFORE sort (line 1407) and BEFORE pagination (line 1418). This is correct — chain movies are never included in the paginated result.
**Warning signs:** If filter moves after pagination in a future refactor, chain movies would reappear.

### Pitfall 3: Frontend CSV Parser — CRLF Line Endings
**What goes wrong:** `text.trim().split("\n")` at line 25 may not handle Windows-style `\r\n` line endings, leaving `\r` at the end of fields.
**Why it happens:** FileReader on Windows may produce CRLF. The current code uses `.replace(/^"|"$/g, "")` but doesn't strip `\r`.
**How to avoid:** Add `\r` stripping when splitting lines, or normalize: `text.replace(/\r\n/g, "\n").trim().split("\n")`.
**Note:** This is a related cleanup worth doing in the same fix as the quoted-field parser.

### Pitfall 4: Deploy Required for All Fixes
**What goes wrong:** Plans 04-08 and 04-09 code is already in the repo. Writing a new plan with the same backend code produces no-op redundancy.
**How to avoid:** The planner must NOT rewrite the BUG-01 exclusion logic or BUG-03 fallback trigger — these changes exist. The plan should (1) fix BUG-03's early-return issue (small surgical change), (2) fix BUG-02 in the frontend, (3) deploy via Docker rebuild + NAS verification.

## Open Questions

1. **BUG-03 — Does the fallback actually fix the long-chain case?**
   - What we know: The fallback at line 1769 only runs when `len(top5) < 5` and `genre_freq` is non-empty. It is unreachable from the `if not eligible_actors: return []` early exit.
   - What's unclear: Whether the "not resolving on NAS" report means the fallback ran but produced bad results, or whether the early exit prevented it from running at all.
   - Recommendation: The planner should fix the early-return masking issue (add fallback path when `eligible_actors` is empty) before deploying. Do not rely on NAS observation to diagnose this — the code path analysis is definitive.

2. **BUG-02 — Option A vs Option B (CSV fix vs .xlsx support)**
   - What we know: UI-SPEC recommends Option B (additive — keep .csv, add .xlsx). But Option A (fix the frontend parser) requires zero new dependencies and handles the round-trip case.
   - What's unclear: Whether users actually have comma-in-title data in .csv files they want to import, or whether the failure was only observed with specific titles.
   - Recommendation: Option A (fix `parseCSV`) is strictly better as a default — it fixes the root cause. Option B can be layered on top if desired (add `openpyxl`, adjust endpoint) but is not required for correctness.

3. **Are there other locations that call CSV parsing that need fixing?**
   - What we know: The only CSV parsing in the frontend is `parseCSV()` in `GameLobby.tsx`. The backend export uses Python `csv.writer` (correct). The backend `import-csv` endpoint receives pre-parsed rows from the frontend (the `CSVRow` Pydantic model), so it never parses CSV text itself.
   - Recommendation: Only `parseCSV()` in `GameLobby.tsx` needs fixing.
